# 012 — `ref Parameter`

## 1. `ref` neden var?

C# parametreleri varsayılan olarak **by-value** aktarılır. Bu kural value type ve reference type için ortaktır.

- `int` gönderildiğinde sayısal değer kopyalanır.
- `Player` gönderildiğinde object kopyalanmaz; caller variable'ın taşıdığı **reference value** kopyalanır.

Bu varsayılan davranış metot sınırlarında izolasyon sağlar: callee kendi parameter variable'ını yeniden atadığında caller variable değişmez.

Bazen problem tam tersidir. Metodun yalnızca bir değeri kullanması değil, caller tarafındaki gerçek variable storage'ını okuyup yazması gerekir. `ref` bu ihtiyacı açıkça ifade eder.

## 2. Hangi problemi çözer?

Aşağıdaki metot caller variable'ı değiştiremez:

```csharp
static void Replace(Player p)
{
    p = new Player { Name = "Mehmet" };
}
```

Çağrı:

```csharp
Player player = new() { Name = "Ali" };
Replace(player);
```

Normal aktarımda iki ayrı variable oluşur:

```text
Caller variable:    player ──► Player #1
Parameter variable: p      ──► Player #1
```

`p = new Player()` yalnızca parameter variable'a yeni reference value yazar:

```text
Caller variable:    player ──► Player #1
Parameter variable: p      ──► Player #2
```

Caller variable'ın kendisini değiştirmek için:

```csharp
static void Replace(ref Player p)
{
    p = new Player { Name = "Mehmet" };
}
```

Bu kez callee bağımsız bir reference value kopyası üzerinde değil, caller variable'ın storage location'ı üzerinde çalışır.

## 3. `ref` olmasaydı ne yapmak zorunda kalırdık?

Çoğu durumda dönüş değeri kullanılabilir:

```csharp
static Player Replace(Player p)
{
    return new Player { Name = "Mehmet" };
}

player = Replace(player);
```

Bu yaklaşım genellikle daha açık ve daha az yan etkilidir. Ancak bazı senaryolarda storage identity veya birden fazla variable üzerinde yerinde işlem önemlidir:

```csharp
static void Swap<T>(ref T left, ref T right)
{
    T temp = left;
    left = right;
    right = temp;
}
```

`ref` olmadan tuple döndürmek de mümkündür:

```csharp
static (T Left, T Right) Swap<T>(T left, T right)
    => (right, left);
```

İki yaklaşım aynı semantiği ifade etmez:

- `ref`, mevcut caller storage'larını yerinde değiştirir.
- tuple dönüşü yeni değerler üretir ve caller'ın açıkça atama yapmasını ister.

## 4. Caller Variable ve Parameter Variable

### Normal parameter

```csharp
static void Change(Player p)
{
    p = new Player();
}
```

`player` ve `p` iki ayrı variable'dır. Başlangıçta aynı reference value'yu taşıyabilirler.

### `ref` parameter

```csharp
static void Change(ref Player p)
{
    p = new Player();
}
```

C# kaynak kodunda `p` bir parameter olarak görünür; fakat parameter'ın semantiği caller variable'ın storage'ına by-reference erişimdir. `p` üzerinden yapılan atama caller storage'ına yazılır.

## 5. Managed Pointer nedir?

Managed pointer, CLR type sisteminde bir managed storage location'a by-reference erişimi temsil eder.

C#:

```csharp
static void Increment(ref int value)
{
    value++;
}
```

IL signature kavramsal olarak:

```text
void Increment(int32& value)
```

Buradaki `&`, `value` parametresinin `int32` değer kopyası değil, `int32` storage'ına managed byref olduğunu gösterir.

Managed pointer:

- bir local variable'ı,
- bir argument slot'unu,
- bir object field'ını,
- bir array element'ini

hedefleyebilir.

Sıradan native pointer ile aynı model değildir. CLR ve JIT bu referansın managed storage'a ait olduğunu bilir; GC ile ilgili güvenlik ve yaşam süresi kuralları uygulanır.

## 6. Compiler ve IL bunu nasıl uygular?

Caller'ın gönderdiği storage türüne göre farklı instruction'lar kullanılabilir:

| IL instruction | Amaç |
|---|---|
| `ldloca` | Local variable'ın managed address'ini yükler. |
| `ldarga` | Argument slot'un managed address'ini yükler. |
| `ldflda` | Object/static field'ın managed address'ini yükler. |
| `ldelema` | Array element'inin managed address'ini yükler. |

Örnek:

```csharp
int number = 5;
Increment(ref number);
```

Kavramsal akış:

```text
ldloca number
call void Increment(int32&)
```

Metot içinde değeri okumak ve yazmak için managed pointer dereference edilir. Kullanılacak kesin IL instruction dizisi compiler optimizasyonuna ve koda bağlıdır; önemli olan method signature'ın `T&` olması ve caller'ın bir storage address'i sağlamasıdır.

### `ldarga` hakkında kritik düzeltme

Her `ref` çağrısında `ldarga` kullanılmaz.

- Local variable için `ldloca`
- Mevcut method argument'i için `ldarga`
- Field için `ldflda`
- Array element'i için `ldelema`

kullanılması beklenebilir.

Bu ayrım, `ref`in “bir object adresi” değil, farklı managed storage location'ların ortak byref semantiği olduğunu gösterir.

## 7. JIT ve GC perspektifi

JIT, `T&` bilgisini ordinary value'dan farklı ele alır. Byref'in hangi storage'a eriştiğini ve yaşam süresi sınırlarını gözetir.

Managed pointer heap içindeki bir field veya array element'ini hedefliyorsa GC compacting sırasında object'in fiziksel konumu değişebilir. Bu nedenle managed pointer'ı kalıcı, GC'den bağımsız bir native adres gibi düşünmek yanlış olur. CLR/JIT byref bilgisini GC ile uyumlu biçimde yönetir.

`ref` kullanmak tek başına:

- boxing oluşturmaz,
- object allocation yapmaz,
- değeri zorunlu olarak heap'e taşımaz.

## 8. Stack diyagramı

### Normal reference type parameter

```text
Main Stack Frame                  Change Stack Frame
┌──────────────────────┐          ┌──────────────────────┐
│ player               │          │ p                    │
│ reference value ─────┼────┐     │ copied ref value ───┼───┐
└──────────────────────┘    │     └──────────────────────┘   │
                            └──────────────┬──────────────────┘
                                           ▼
                                      Heap: Player #1
```

`p = new Player()` yalnızca callee frame'indeki `p` slot'unu değiştirir.

### `ref` parameter

```text
Main Stack Frame                  Change Stack Frame
┌──────────────────────┐          ┌─────────────────────────────┐
│ player storage       │◄─────────│ p : managed pointer (T&)   │
│ reference value      │          └─────────────────────────────┘
└──────────┬───────────┘
           ▼
      Heap: Player #1
```

`p = new Player()` managed pointer'ın hedefi olan `player storage` içine yeni reference value yazar.

## 9. Heap diyagramı

Başlangıç:

```text
player ─────► Player #1 { Name = "Ali" }
```

Object mutation:

```csharp
p.Name = "Veli";
```

Sonuç:

```text
player ─────► Player #1 { Name = "Veli" }
```

Reference reassignment (`ref` ile):

```csharp
p = new Player { Name = "Mehmet" };
```

Sonuç:

```text
player ─────► Player #2 { Name = "Mehmet" }

Player #1 { Name = "Veli" }
└── Başka reachable reference yoksa ileride GC adayı olabilir.
```

## 12. Value Type ile `ref`

`ref` yalnızca reference type'lar için değildir:

```csharp
static void Increment(ref int value)
{
    value++;
}
```

```text
Caller storage: number = 5
Managed pointer: value ──► number storage
Write: number = 6
```

Ortak nokta storage'a erişimdir:

- `ref int`: caller storage içinde doğrudan `int` değeri bulunur.
- `ref Player`: caller storage içinde bir reference value bulunur.

## 13. Gerçek projelerde kullanım

### Yerinde swap ve algoritmalar

```csharp
static void Swap<T>(ref T left, ref T right)
{
    (left, right) = (right, left);
}
```

### Atomik operasyonlar

`Interlocked` gibi düşük seviyeli eşzamanlılık API'leri, güncellenecek storage'ı açıkça belirtmek için `ref` kullanır.

```csharp
Interlocked.Increment(ref counter);
```

### Struct üzerinde yerinde güncelleme

Kopyalama veya yeni değer döndürme yerine mevcut struct storage'ını güncellemek gereken düşük seviyeli kodlarda kullanılabilir.

### Ref return / ref local altyapısı

Array, span veya performans odaklı collection kodlarında element storage'ına doğrudan erişim gereken tasarımların temelini oluşturur.

### Native interop

Bazı P/Invoke sözleşmelerinde native API'nin pointer/out parameter semantiğini modellemek için `ref` veya `out` gerekir.

## 14. Framework örnekleri

Genel amaçlı framework API'leri `ref`i sınırlı kullanır çünkü caller state'ini doğrudan değiştiren sözleşmeler daha zor okunur.

Düşük seviyeli .NET API'lerinde daha yaygındır:

- `Interlocked` operasyonları
- collection/array element'ine byref erişim sağlayan performans API'leri
- `Span<T>` ve ref-safe context ile bağlantılı kodlar
- interop katmanları

ASP.NET Core veya Entity Framework Core gibi uygulama framework'lerinde ana genişletme modeli genellikle DI, return value, middleware, options veya mutable object state üzerinden kurulur; `ref` çoğu public workflow için ilk tercih değildir.

## 15. Reflection `ref` parameter'ı nasıl görür?

```csharp
MethodInfo method = typeof(Sample).GetMethod(nameof(Sample.Change))!;
ParameterInfo parameter = method.GetParameters()[0];

bool isByRef = parameter.ParameterType.IsByRef;
Type? elementType = parameter.ParameterType.GetElementType();
```

`ref Player` için:

```text
IsByRef    = true
ElementType = Player
```

Reflection burada method metadata'sını okur. `ref` yalnızca C# syntax bilgisi olarak kaybolmaz; runtime type signature'ın parçasıdır.

## 16. Kullanılmaması gereken durumlar

`ref` şu durumlarda ilk tercih olmamalıdır:

- Metot tek bir yeni değer üretiyorsa dönüş değeri daha açıksa
- Caller state'inin gizlice değişmesi API'yi zor anlaşılır yapıyorsa
- Domain invariant'ları field/property katmanında korunmalıysa
- Alias sayısı arttığı için hata analizi zorlaşıyorsa
- Performans gerekçesi ölçülmemişse
- Immutable tasarım daha güvenli ve yeterliyse

Kötü tasarım sinyali:

```csharp
void Calculate(ref int total, ref int tax, ref int discount, ref string message)
```

Daha açık alternatif:

```csharp
CalculationResult Calculate(Order order)
```

## 17. Performans notları

`ref`, özellikle büyük struct'ların kopyalanmasını azaltabilecek bir araçtır; ancak performans sonucu bağlama bağlıdır.

Dikkat edilmesi gerekenler:

- Küçük value type'larda kopyalama çoğu zaman ucuzdur.
- Byref erişim alias analysis ve optimizasyon kararlarını etkileyebilir.
- Reference type için normal parametrede zaten object kopyalanmaz; yalnızca reference value kopyalanır.
- `ref Player`ı “object kopyasını engellemek” için kullanmak yanlış gerekçedir.
- Ölçüm yapılmadan public API karmaşıklığı artırılmamalıdır.
- Salt okunur büyük struct aktarımı problemi `in` konusunun temel motivasyonudur.

## 18. Alternatifler ve tasarım tercihleri

### Return value

```csharp
player = Replace(player);
```

Tek sonuç için genellikle en açık alternatiftir.

### Tuple

```csharp
(left, right) = Swap(left, right);
```

Birden fazla yeni değer üretimini açıkça gösterir.

### Result object

```csharp
CalculationResult result = calculator.Calculate(order);
```

İsimlendirilmiş sonuçlar, validasyon ve gelecekte genişleme için uygundur.

### Mutable object

Object state zaten paylaşılabiliyorsa member mutation yeterli olabilir; ancak invariant ve thread-safety değerlendirilmelidir.

### `out`

Callee'nin caller storage'ına bir sonuç yazması ve giriş değerini okumak zorunda olmaması problemi için tasarlanmıştır. Sonraki konudur.

### `in`

By-reference aktarım yaparken callee'nin değeri değiştirmesini istemediğimiz durumda kullanılır. Özellikle büyük struct'lar için motivasyonu vardır.

## 19. Özet

- C# parametreleri varsayılan olarak by-value aktarılır.
- Reference type parametrede object değil, reference value kopyalanır.
- Caller variable ve parameter variable normal aktarımda ayrı storage'lardır.
- Object mutation, object state'ini değiştirir.
- Reference reassignment, variable'ın taşıdığı reference value'yu değiştirir.
- `ref`, caller variable'ın storage location'ına managed pointer semantiğiyle erişim sağlar.
- CLR metadata'sında `ref T`, `T&` olarak temsil edilir.
- `ldloca`, `ldarga`, `ldflda` ve `ldelema` farklı storage türleri için managed address üretir.
- `ref` boxing değildir ve zorunlu heap allocation oluşturmaz.
- Kullanımı semantik ihtiyaçla gerekçelendirilmelidir; yalnızca “daha hızlı olabilir” varsayımı yeterli değildir.
- Sonraki konu: **`out parameter`**.
